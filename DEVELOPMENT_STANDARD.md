# Odoo 团队开发规范

> 本文档是完整的开发规范参考，同时作为 PR Review Action 的审查依据。
> 日常 AI 开发请使用精简版 `Agent.md`。

版本：v2.0
基线：Odoo 18.0

---

## 1. 这份规范的目标

降低五类成本：

1. **升级成本**：避免侵入式改动导致跨版本升级困难
2. **规模成本**：避免 N+1、锁冲突、计算雪崩
3. **协作成本**：让人和 AI 产出的代码结构一致、可审查
4. **运维成本**：权限、安全、多公司、日志从第一天进入开发
5. **语义成本**：避免同一业务概念被重复建字段、重复建状态

---

## 2. 架构原则

### 2.1 版本无关架构

业务逻辑不应绑死在特定 Odoo 版本的实现细节上。

```
┌──────────────────────────────┐
│  业务规则层（版本无关）         │  纯判断、纯计算、纯校验
│  "超过阈值需要审批"            │  提取为独立方法
├──────────────────────────────┤
│  ORM 适配层（版本相关）         │  create/write/search/xpath
│  升级时主要改这一层             │  保持薄包裹
├──────────────────────────────┤
│  Odoo 框架                    │
└──────────────────────────────┘
```

**落地方式：**

- 业务判断提成 `_needs_*()` / `_check_*()` / `_get_*()` 独立方法
- ORM 操作集中在 `_do_*()` / `_prepare_*()` / `action_*()` 中
- 复杂流程拆成可 override 的小方法链，而不是一个大方法

### 2.2 一个业务概念只能有一个权威来源

同一个业务概念（审批状态、风险等级、外部唯一键等），必须先确定：

- 权威字段在哪个模型（Canonical Field）
- 权威模块是谁（Owner Module）
- 谁负责写入
- 谁只负责读取或投影

其他模块需要该语义时，用 `related` / `compute` / helper 方法读取，不另建同义字段。

### 2.3 先复用，再新增

新增模型、字段、配置、状态、定时任务之前，必须先检查：

1. Odoo 标准能力是否已有可直接复用的对象
2. 团队已有自定义模块是否已有同语义对象
3. 是否可以通过 `related`、compute 投影、link module、配置映射解决

### 2.4 升级优先于短平快

禁止直接修改 Odoo 官方源码。所有定制通过标准扩展点完成：模型继承、视图继承（xpath）、安全规则、数据文件、控制器扩展、JS/Owl patch。

### 2.5 批量优先于单条思维

Odoo 的工作模型是 recordset。设计方法时默认支持批量，只有天然单条的操作（按钮 action、向导确认）才 `ensure_one()`。

### 2.6 安全默认收紧

不以菜单隐藏、按钮隐藏、前端 invisible 作为安全边界。安全控制必须以后端 ACL、Record Rule、字段 groups、显式校验为准。

### 2.7 多公司默认存在

除非确定是单公司部署，否则默认按多公司设计。

## 3. 模块架构

### 3.1 一个模块一个职责

- 业务域模块：销售、采购、仓储、财务、主数据
- 集成模块：对接外部系统（电商、WMS、MES）
- 规则模块：审批、风控、价格策略
- 报表模块：BI、标签、报表增强

需求涉及多个域时，拆成多个模块通过 `depends` 衔接。

### 3.2 模块分层

```
核心共享层   通用语义、主数据扩展、mixin、共享配置
    ↑
业务域层    销售、采购、库存、财务等单域能力
    ↑
联动层      连接多个业务域的 link module
    ↑
集成层      对接外部系统
    ↑
项目层      客户/项目专属定制
```

- 上层可以依赖下层，反过来不行
- 共享字段放核心共享层或业务域核心模块
- 两个模块间的组合行为放联动层（link module），不塞进任一一方
- 客户专属逻辑不污染共享模块

### 3.3 Link Module 优先于硬耦合

模块 A 和模块 B 各自独立，组合安装后才需要额外逻辑时，新建 link module：

```
myco_sale_stock_link     # 负责销售-库存联动
myco_purchase_quality_link   # 负责采购-质检联动
```

Link module 负责：组合字段、联动约束、联动按钮、联动定时任务。

### 3.4 共享对象必须有 Owner Module

每个被多模块复用的对象（字段、配置、mixin、状态枚举）必须指定 owner module。其他模块只能扩展或消费，不能平行复制。

### 3.5 推荐目录结构

```
my_module/
├── __init__.py
├── __manifest__.py
├── models/
├── views/
├── security/
│   ├── ir.model.access.csv
│   └── my_module_security.xml
├── data/
├── demo/
├── wizard/
├── report/
├── controllers/
├── static/
├── tests/
└── migrations/
```

### 3.6 `__manifest__.py` 规则

- `depends` 显式声明，禁止靠安装顺序凑巧能跑
- `data` 按依赖顺序：安全 → 基础数据 → 视图 → 菜单/动作
- `demo` 仅演示数据
- `version` 语义化版本
- `license` 必填

### 3.7 不允许隐式跨模块耦合

- 不在模块 A 中假设模块 B 的字段存在，却不在 depends 声明 B
- 不通过字符串或配置偷偷引用另一个模块的未公开实现
- 不在 XML 里继承另一个模块视图却不声明依赖

## 4. 代码规范

### 4.1 模型类内部排列顺序

见 `Agent.md` 第三节。严格按以下顺序：

元属性 → 字段 → SQL 约束 → CRUD 覆写 → compute → onchange → constrains → action_* → _private → _cron

### 4.2 Import 顺序

标准库 → 第三方 → Odoo 框架 → 本模块。详见 `Agent.md` 第九节。

### 4.3 CRUD 覆写规则

- `create` 使用 `@api.model_create_multi`，支持 `vals_list`
- `super()` 只调用一次
- 逻辑拆到私有 helper
- 返回值保持与父类一致

```python
@api.model_create_multi
def create(self, vals_list):
    vals_list = self._prepare_create_vals(vals_list)
    records = super().create(vals_list)
    records._post_create_hook()
    return records
```

### 4.4 计算字段

- `@api.depends` 精确声明，不过宽也不漏
- 参与搜索、过滤、分组、排序、高频列表展示的计算字段优先 `store=True`
- 避免计算雪崩：`write()` 修改高扇出依赖字段时评估影响

### 4.5 约束优先级

1. 数据库层约束（NOT NULL、UNIQUE）：绝对数据规则
2. `_sql_constraints`：稳定且简单的约束
3. `@api.constrains`：跨字段、带业务语义的约束
4. `onchange`：仅用于改善录入体验，不作为业务约束

### 4.6 `sudo()` 使用规则

- 只对最小 recordset、最小操作使用
- 使用前注释为什么必须绕过权限
- `sudo()` 后仍校验业务合法性
- 禁止 `return self.sudo().entire_method()`

### 4.7 `with_context()` 使用规则

- 只传最小必要 key
- 自定义 context key 用模块前缀命名
- 注释影响范围
- 禁止用 context 偷渡核心业务参数

### 4.8 Raw SQL

仅在 ORM 无法满足且有明确性能证据时使用。同时满足：

- 参数化查询
- 注释为什么不能用 ORM
- 明确 `flush()` / `invalidate_cache()` / `modified()` 处理
- 有测试覆盖结果一致性

### 4.9 事务

禁止 `self.env.cr.commit()` / `rollback()`。唯一例外：显式创建独立 cursor。

### 4.10 Mixin 设计

- 2 个以上不同 `_name` 的模型需要相同能力时考虑 mixin
- 用 `models.AbstractModel`，`_name` 以 `.mixin` 结尾
- 字段命名带前缀避免与消费者冲突
- 不定义消费者应该覆写的默认值

### 4.11 Wizard 设计

- 使用 `TransientModel`，不存长期数据
- 负责收集输入和校验，业务逻辑调用目标模型的方法
- 支持从 `active_ids` 获取批量目标
- action 方法返回正确的窗口动作（关闭/跳转）

### 4.12 Controller 设计

- 路由带模块前缀：`/myco_module/...`
- 入参校验在 controller 层完成
- 业务逻辑调用 model 方法，不在 controller 中直接操作 ORM
- 返回格式统一

### 4.13 状态机设计

- Selection 值用英文词，不用数字
- 状态变更只通过 `action_*` 方法，禁止直接 `write({'state': ...})`
- 转换矩阵集中定义
- 变更方法内部校验前置状态

## 5. 命名规范

详见 `Agent.md` 第四节。此处补充：

### 5.1 标准模型上的自定义字段

禁止使用容易和官方未来字段冲突的裸名（`status`、`type`、`code`）。使用业务前缀或模块前缀。

不推荐机械地全部加 `x_` 前缀——这会和 Studio 手工字段混淆。

### 5.2 字段分类

| 类型 | 说明 | 规则 |
|------|------|------|
| Canonical | 业务权威字段，唯一事实来源 | 每个语义只能有一个 |
| Projection | 为视图/报表/集成派生的投影 | 优先用 `related` / `compute` |
| Transitional | 迁移期临时兼容 | 必须有淘汰计划 |

### 5.3 重复字段的坏味道

- `sale.order` 上有 `approval_state`，另一个模块又加 `x_approval_status`
- `stock.picking` 上有 `wave_no`，第三个模块又加 `batch_code` 表示同一件事
- 不同模块各存一份"是否已同步"，没有统一语义

## 6. XML ID、数据文件与可升级性

### 6.1 所有可引用对象必须有稳定 XML ID

视图、动作、菜单、安全组、记录规则、定时任务、序列、邮件模板、报表。

### 6.2 XML ID 命名

- 视图：`{model_name}_view_{type}`
- 动作：`{model_name}_action`
- 菜单：`{model_name}_menu`
- 安全组：`{module}_group_{role}`
- 记录规则：`{model_name}_rule_{group}`

### 6.3 永远不硬编码数据库 ID

```python
# ✗
self.env["res.partner"].browse(7)
# ✓
self.env.ref("base.main_company")
```

### 6.4 `noupdate` 只用于真正不希望升级覆盖的数据

初始主数据、系统参数默认值、需手工维护的配置。不要因为怕覆盖就把所有 XML 放进 `noupdate="1"`。

## 7. 安全规范

### 7.1 ACL

每个自定义模型必须有 `ir.model.access.csv`。即便只有管理员能看，也要显式写。

### 7.2 Record Rule

Odoo 记录规则是"默认放行"。没写规则不等于安全，如果 ACL 放开了而没有规则，访问很可能是允许的。

### 7.3 字段级安全

敏感字段用 `groups=`：成本价、毛利率、审批意见、财务字段、风险评分。

### 7.4 公有方法默认可被 RPC 调用

不以下划线开头的方法，都是潜在外部入口。必须在方法内部校验状态、权限、公司范围。

### 7.5 前端隐藏不是权限控制

`groups` 控制按钮显示、`invisible` 隐藏字段、菜单不给用户看——这些都不能替代后端校验。

---

## 8. 多公司规范

- 公司相关模型必须有 `company_id` + `check_company=True`
- 公司相关计算字段必须声明 `depends_context('company')`
- 禁止跨公司脏写（A 公司订单选 B 公司仓库/科目）
- `sudo()` 跨公司跑批时必须限定 company scope，配合 `with_company()`

---

## 9. 视图规范

- 永远继承视图，不复制官方整份 XML
- 用 `xpath` 做外科手术式修改
- 锚点按字段 `name`、按钮 `name` 定位
- 复杂显示条件沉淀为后端字段（`can_approve`、`show_warning`）
- 按钮后端方法必须再次校验状态和权限
- 明确使用当前 Odoo 版本的视图表达式语法

---

## 10. 测试规范

### 10.1 必须有测试的改动

新模型/流程、权限变更、价格/库存/会计/审批逻辑、计算字段依赖变更、多公司逻辑、定时任务、集成同步。

### 10.2 至少覆盖的场景

1. 正常路径
2. 批量路径
3. 权限/安全路径
4. 多公司路径
5. 回归路径
6. 复用路径：消费模块读取的是 canonical source

### 10.3 测试类型选择

- `SavepointCase`：大多数模型逻辑测试
- `TransactionCase`：需完整事务隔离
- `HttpCase`：页面或控制器流程

### 10.4 性能断言

性能敏感逻辑补 `assertQueryCount()`，防止 batch 退化成 N+1。

### 10.5 AI 产出的测试要求

AI 生成业务代码必须附测试。以下极小变更除外：标签文案、纯样式、无逻辑帮助文本。

## 11. 数据、配置与初始化

### 11.1 业务参数禁止硬编码

阈值金额、仓库名、审批人 ID、公司 ID、税率——都不得写死在代码中。落到 `res.config.settings` / `ir.config_parameter` / 独立配置模型。

### 11.2 区分数据类型

- 主数据：安装时初始化，可被业务维护
- 配置数据：系统依赖，变更要谨慎
- 演示数据：仅 demo
- 测试数据：仅服务测试，不混进正式数据文件

### 11.3 定时任务必须幂等

- 重复执行安全
- 失败可重试
- 不重复创建单据/消息/附件
- 有批次边界与日志

---

## 12. 集成与外部接口

### 12.1 事务边界

核心写库先完成，外呼改为异步任务或可重试机制。需要同步时，明确补偿策略。

### 12.2 接口幂等

有外部唯一键、有去重策略、重复推送不造成重复数据、错误日志足以排障。

---

## 13. 日志与可观测性

- `UserError` / `ValidationError` 说明发生了什么、如何修正，不暴露技术堆栈
- 关键链路（同步、审批、批处理、自动过账）记结构化日志
- 批量处理不要对每条记录都 `message_post`，除非业务确实需要审计

## 14. Git 与协作规范

### 14.1 分支策略

- `main`：生产稳定
- `staging`：预发布/集成测试
- `feature/*`：新功能
- `fix/*`：缺陷修复

使用 Odoo 风格前缀，详见 `COMMIT_CONVENTION.md`。

### 14.3 PR 要求

提交 PR 时必须填写 PR 模板中的所有字段（见 `.github/pull_request_template.md`）。

## 15. 废弃协议

### 15.1 废弃标记

废弃的字段/方法必须：

- 代码中加 `_deprecated` 标记或 `warnings.warn()`
- 字段 `help` 注明废弃原因和替代方案
- 在 commit message 中用 `[DEP]` 前缀

### 15.2 保留期

至少保留一个完整版本周期再移除。

### 15.3 移除要求

移除时必须有迁移脚本处理残余数据。

### 15.4 过渡字段退出机制

迁移期临时字段必须同时记录：

- 服务于哪个版本迁移
- 预计保留到哪个里程碑
- 删除条件
- 删除时的数据迁移方案

---

## 16. 重构指南

### 16.1 重构三步法

禁止大爆炸式重构。采用：

1. **先加新的**：创建新结构，新旧并存
2. **再迁移**：逐步把调用方迁移到新结构
3. **后删旧的**：确认无调用后删除旧代码 + 数据迁移脚本

### 16.2 重构前提

- 确认目标模块测试覆盖率，不足则先补测试
- 每个重构 PR 只做一类事（拆方法 / 提 mixin / 移字段），不混功能变更
- 附 `assertQueryCount` 证明性能未退化

### 16.3 常见重构场景

**合并重复字段**：

1. 确定 canonical source
2. 新增 related/compute 把消费方指向 canonical
3. 数据迁移脚本同步历史数据
4. 移除冗余字段

**拆分过大模块**：

1. 识别职责边界
2. 提取新模块，旧模块 depends 新模块
3. 逐步迁移代码
4. 最终删除旧模块中的冗余代码

**提取 Mixin**：

1. 识别多模型共享的逻辑
2. 创建 AbstractModel mixin
3. 改写消费模型 `_inherit` mixin
4. 删除各模型中的重复代码

**分离业务逻辑层**：

1. 识别大方法中的业务规则
2. 提取为 `_check_*` / `_needs_*` / `_get_*` 独立方法
3. 原方法改为调用小方法的流程编排
4. 验证行为不变

## 17. 代码评审清单

PR Review（人工或 AI）至少检查以下 15 项：

1. 是否通过标准扩展点实现，而非侵入式修改
2. 方法语义是否清晰（单条 vs 批量）
3. 是否存在循环内 search / read_group / 高成本读取
4. 是否误把 onchange 当业务约束
5. 是否新增了字段，有没有先做复用检查
6. 新增字段是否明确 owner module 和 canonical source
7. 是否引入 sudo()，边界是否合理
8. 是否考虑多公司
9. 是否有 ACL、Record Rule、字段 groups
10. 是否使用稳定 XML ID 和稳定 xpath 锚点
11. 是否需要 store=True、索引、迁移脚本
12. 是否补了测试
13. 业务逻辑是否与 ORM 操作分离（可重构性）
14. 关键方法是否拆成可 override 的小方法
15. 状态变更是否通过 action_* 方法而非直接 write

## 18. AI 任务模板

### 18.1 功能开发

```
Odoo 版本：18.0
目标模块：
业务场景：
期望行为：
涉及模型：
是否新增字段：如是，为什么不能复用
禁止修改范围：

交付要求：
1. 使用标准扩展点
2. 先确认模型/字段/XML ID 真实存在
3. 最小 diff
4. 说明影响范围和迁移需求
5. 附测试
```

### 18.2 Code Review

```
从 Odoo 团队视角审查以下改动。
对照 DEVELOPMENT_STANDARD.md 第 17 节的 15 项清单逐项检查。

输出：
- Blocker：必须修才能合并
- Warning：建议修改
- Info：提示信息
```

### 18.3 重构

```
重构以下代码，必须遵守：

3. 不改变字段语义
```

## 19. 一句话总结

AI 驱动的 Odoo 团队，不是让 AI 多写代码，而是把扩展边界、批量思维、安全模型、升级约束和测试纪律固化下来，让 AI 在边界内稳定产出，让将来的重构有安全网可依赖。
