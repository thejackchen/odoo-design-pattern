# 架构愿景：让业务知识不被锁死在任何软件里

> 本文用尽量简单的语言，解释我们为什么要做这件事、怎么一步步做。
> 不需要技术背景也能读懂。

---

## 一、我们到底在解决什么问题

假设你开了一家公司，业务规则是：

> "订单金额超过 10 万，需要经理审批。"

这条规则跟你用什么软件没有关系。你用 Odoo 要这么做，用 ERPNext 也要这么做，用 Excel 也要这么做，甚至回到纸质时代也要这么做。

但现实是，这条规则被**写死在 Odoo 的代码里**了：

```python
if self.amount_total > 100000:
    raise UserError("需要审批")
```

这行代码绑定了 Odoo 的写法（`self.amount_total`、`UserError`）。哪天 Odoo 升级了 API，或者你想换个软件，这行代码就废了。但**你的业务规则没变**。

### 被锁住的不只是"规则"

很多人以为"业务知识"就是几条 if-else 规则。其实远不止：

| 业务知识类型 | 举例 | 通常藏在哪里 |
|---|---|---|
| **业务规则** | 大额订单需要审批 | Python 方法的 if 判断里 |
| **数据模型** | 客户有等级，等级影响折扣 | models.py 的字段定义里 |
| **状态流转** | 草稿→确认→审批→完成 | Selection 字段 + action_* 方法里 |
| **权限设计** | 销售员不能看成本价 | ir.model.access.csv + record rule XML 里 |
| **审批流程** | 三级审批：主管→经理→总监 | 多个 wizard + 多个状态字段里 |
| **报表逻辑** | 月度销售按区域汇总 | report 模板 + SQL 查询里 |
| **集成映射** | 我们的"客户编号"= 对方系统的"BP Code" | 接口代码的字段映射字典里 |
| **计算公式** | 折扣 = 基础折扣 × 客户等级系数 × 季节系数 | compute 方法的 Python 表达式里 |
| **校验规则** | 发货数量不能超过库存数量 | constrains 装饰器里 |
| **通知规则** | 审批通过后自动发邮件给销售员 | mail.template + 各种触发点里 |

**这些加在一起，就是你公司花了几年、甚至十几年积累下来的"运营智慧"。** 它们目前全部散落在 Odoo 的各种代码文件里，跟 Odoo 的语法、API、框架深度绑定。

我们的目标是：**把这些业务知识从软件实现中分离出来，用任何人都能看懂的格式保存。**

## 二、先搞懂 Odoo 的工作方式（为什么升级会痛）

### Odoo 的"继承"是什么意思

想象 Odoo 官方给了你一份**菜谱**（比如"销售订单"的功能）。

你想在这个菜谱上加一步——"如果金额超过 10 万，先让经理尝一口"。

Odoo 的做法是：**你直接在人家的菜谱上涂改**。

```
Odoo 官方菜谱《销售订单》：
  第 1 步：客户下单 ✓
  第 2 步：确认订单 ← 你在这里插了一段话："先检查金额，超过10万要审批"
  第 3 步：发货 ✓
```

这就是 `_inherit`——你不是写一份新菜谱，你是**在别人的菜谱上加批注**。

这样做的好处是简单直接，但问题是：

- **官方改了菜谱，你的批注可能就对不上了。** 比如官方把"第 2 步"拆成了两步，你的批注还指向老的"第 2 步"，就会错位。
- **多个人同时在一份菜谱上涂改，互相打架。** 张三在第 2 步加了审批，李四也在第 2 步加了风控检查，两个人的批注可能冲突。
- **你的业务知识和菜谱的格式混在一起。** 你写的"金额超过 10 万要审批"不是一条独立的规则，而是嵌入在菜谱某一步的某一行里。将来想把这条规则移到别的地方，或者用到别的系统里，得把它从菜谱里抠出来。

**Odoo 每次大版本升级，本质上就是官方把菜谱重写了一遍。** 你之前涂改的那些批注，全部要检查一遍、对齐一遍、修一遍。模块越多越痛。

### Odoo 的视图继承也是同样的问题

Odoo 的页面（视图）也是用类似的方式修改的。官方给了你一个表单页面，你想加一个字段，不是从头画一个新页面，而是说"在某某字段后面加一个新字段"。

```xml
<xpath expr="//field[@name='partner_id']" position="after">
    <field name="approval_state"/>
</xpath>
```

这相当于说"找到菜谱里写着'partner_id'的那一行，在它后面加一行"。

问题是：如果官方升级时把 `partner_id` 这个字段移到了别的位置，或者改了名字，你的定位就失效了。

### 升级到底有多痛：一个真实场景

假设你的公司有 15 个自定义模块，现在要从 Odoo 16 升级到 Odoo 18。以下是你会遇到的典型场景：

**场景 1：API 改了名字**

Odoo 17 把 `product.product` 里的 `qty_available` 重命名了一部分逻辑。你有 3 个模块引用了这个字段。找到这些引用不难，但你得确认新版本的行为跟你预期的还一样——这需要读新版源码。

**场景 2：方法签名变了**

Odoo 18 的 `action_confirm` 增加了新参数。你覆写了这个方法，但没有传递新参数，结果官方逻辑的一部分被跳过了——而且这个 bug 不会报错，只是数据算出来不对。可能几个月后才被发现。

**场景 3：视图结构变了**

官方把销售订单的表单页面从 `notebook` 布局改成了 `tab` 布局。你的 xpath 全部定位失败。涉及 8 个模块、20 多个 xpath。每一个都要重新定位。

**场景 4：依赖链爆炸**

模块 A 继承了 `sale.order`，模块 B 也继承了 `sale.order` 并且依赖 A 加的字段，模块 C 继承了模块 B 的方法。升级时 A 改了字段名 → B 要跟着改 → C 的覆写也要改。改一个字段，波及三个模块。

**这些场景加在一起，就是为什么一次大版本升级可能需要几个月时间。** 如果业务知识是独立的，升级时只需要改"翻译层"，不需要逐个模块去扒代码。

### 数据迁移：升级中最容易被忽视的炸弹

代码适配完了不算完，还有数据。Odoo 升级时数据库结构会变化：

- 字段被重命名：你存的 `x_old_name` 数据要迁移到 `x_new_name`
- 字段类型变了：比如 `Char` 变成 `Selection`，旧数据格式不匹配
- 关联关系变了：`Many2one` 变成 `Many2many`，需要拆数据
- 整个模型被拆分或合并

如果你的字段定义散落在 15 个模块里，每个模块都需要单独写迁移脚本。如果字段定义集中在配置文件里，迁移工具可以自动对比新旧版本的配置差异，自动生成迁移脚本。

## 三、别人是怎么做的

### ERPNext：模型是"填表格"，不是"写代码"

ERPNext 是另一个开源 ERP 软件，它的升级比 Odoo 顺滑得多。核心区别有两个：

**区别一：模型是"填表格"，不是"写代码"**

在 Odoo 里，你要定义一个新字段，需要写 Python 代码：

```python
class SaleOrder(models.Model):
    _inherit = 'sale.order'
    approval_state = fields.Selection([('pending', '待审'), ('approved', '已审')])
```

在 ERPNext 里，你不写代码，你**填一个表格**（实际上是一个 JSON 文件）：

```json
{
    "字段名": "approval_state",
    "字段类型": "下拉选择",
    "选项": "待审\n已审"
}
```

区别在哪？

- Odoo 的方式：你写的是**指令**——"请按这个方式创建一个字段"。代码就是指令，指令依赖于软件的语法。语法变了，指令就废了。
- ERPNext 的方式：你填的是**描述**——"我需要一个叫 approval_state 的下拉字段"。软件自己去理解这个描述，自动创建。描述是通用的，不依赖于特定语法。

升级的时候，ERPNext 框架会自动对比新旧版本的描述，自动做数据库调整。你不需要手写迁移脚本。

**打个比方：** Odoo 就像你自己动手装修房子（写代码），墙砸了自己修。ERPNext 就像你告诉设计师"我要这样的房子"（填描述），设计师去搞定。设计师换了一批工人（软件升级），但你对房子的需求没变。

**区别二：扩展是"挂钩子"，不是"改别人代码"**

在 Odoo 里，你要在订单确认时加个检查，需要**覆写**官方的方法：

```python
def action_confirm(self):
    self._check_approval()  # 你加的
    return super().action_confirm()  # 调用官方原来的逻辑
```

你的代码和官方代码**编织在一起**了。官方改了 `action_confirm` 的逻辑，你的覆写可能就不对了。

在 ERPNext 里，你不碰官方代码，你在旁边**挂一个钩子**：

```python
# hooks.py
doc_events = {
    "Sales Order": {
        "on_submit": "my_app.check_approval"
    }
}
```

意思是："当销售订单提交的时候，顺便也跑一下我的 check_approval 函数。"

**打个比方：**

- Odoo 的方式：你在别人的生产线上加了一个工位。生产线改了，你的工位可能就接不上了。
- ERPNext 的方式：你在生产线旁边放了一个检测站。生产线上的产品出来后，经过你的检测站。生产线怎么改，你的检测站都能独立运作。

### SAP：干净核心，外挂扩展

**SAP（全球最大的 ERP）** 最近几年推"Clean Core"（干净核心）策略：

> 核心系统保持原装，不要碰。定制的逻辑放在外面，通过 API（接口）连接。

就像你买了一辆车（核心系统），不要改装发动机，改装了以后厂家不保修。你要加功能，就外挂设备（行车记录仪、导航、倒车雷达），通过标准接口连上。车子换代了，外挂设备拔下来插到新车上就行。

SAP 为此定义了严格的分层：

| 层级 | 可以动吗 | 举例 |
|---|---|---|
| 核心代码 | 绝不能动 | SAP 标准功能 |
| 标准扩展点（BAdI） | 推荐使用 | 官方预留的"插座" |
| 自定义 API | 推荐使用 | 你自己暴露的接口 |
| 直接修改核心 | 绝对禁止 | 一碰就不保修 |

**关键启示：** SAP 把"什么能碰、什么不能碰"定义得非常清楚。我们的 Agent.md 里"绝对禁止"清单，本质上也是在做同样的事情。

### Salesforce：一切皆配置

**Salesforce** 走得更远——所有东西都是配置出来的，不是写出来的。你在网页上拖拖拽拽就能定义模型、字段、流程、权限。它存的是你的"配置"，不是你的"代码"。升级时平台自动处理兼容性。

Salesforce 的启示是最激进的：

```
传统方式：人 → 写代码 → 软件执行代码
Salesforce 方式：人 → 在界面上配置 → 平台解释配置 → 软件执行
我们的方向：人/AI → 写 YAML 配置 → 翻译器生成代码 → Odoo 执行代码
```

### Microsoft Dynamics 365：元数据驱动

微软的 Dynamics 365 采用"元数据驱动"的方式：

- 实体（Entity）、字段（Attribute）、关系（Relationship）全部是元数据
- 业务规则用"Business Rule"配置，不写代码
- 工作流用 Power Automate 拖拽搭建
- 升级时，平台自动迁移元数据，不需要手动改代码

**关键启示：** "元数据"就是"描述数据的数据"。我们的 YAML 配置文件，本质上就是 Odoo 模块的元数据。

### 共同点

不管是 ERPNext、SAP、Salesforce 还是 Dynamics，它们做对了同一件事：

> **把"我要什么"和"怎么实现"分开。**

| | Odoo（当前） | 更好的方式 |
|---|---|---|
| 定义字段 | 写 Python 代码 | 填表格 / 写配置文件 |
| 加功能 | 改别人的代码 | 在旁边挂钩子 |
| 存业务规则 | 嵌在代码里 | 独立的规则文件 |
| 存审批流程 | 散落在多个模型里 | 集中的流程配置 |
| 升级时 | 逐行检查代码兼容性 | 自动适配 |
| 换系统时 | 从头再来 | 带着配置走 |

## 四、我们的目标：把业务知识从 Odoo 里"拔"出来

我们不是要重写 Odoo，也不是要换成 ERPNext。我们是要在 Odoo 之上建一个**薄薄的中间层**，让业务知识有一个独立的"家"。

### 最终效果

```
┌────────────────────────────────────┐
│  业务知识层                          │  ← 你和 AI 日常维护的东西
│  "订单超过10万要审批"                 │     普通人能看懂的 YAML 文件
│  "新客户首单要审批"                   │     跟 Odoo 版本无关
│  "审批流：草稿→确认→审批→完成"        │     跟任何软件无关
├────────────────────────────────────┤
│  翻译器 / 适配器                     │  ← 把业务知识翻译成 Odoo 代码
│  读取上面的 YAML                     │     Odoo 升级时只改这一层
│  生成 Odoo 模块代码                  │
├────────────────────────────────────┤
│  Odoo（18.0 / 19.0 / ...）          │  ← 底层软件，可以升级甚至替换
└────────────────────────────────────┘
```

### 业务知识的完整分类

为了有计划地提取业务知识，我们把它分成四大类：

#### 类型 A：业务规则（最优先提取）

条件判断、校验逻辑、审批条件——这些是最容易提取、价值也最高的部分。

```yaml
# business/sales/rules.yaml
rules:
  - name: 大额订单审批
    trigger: 订单确认时
    condition: 订单金额 > 公司审批阈值
    action: 要求审批
    审批人: 销售经理

  - name: 新客户风控
    trigger: 订单确认时
    condition: 客户是新客户 且 是首次下单
    action: 要求审批
    审批人: 销售总监

  - name: 发货数量校验
    trigger: 发货单确认时
    condition: 发货数量 > 库存数量
    action: 阻止并提示
    提示信息: 发货数量不能超过库存数量
```

#### 类型 B：状态流转（第二优先）

业务对象的生命周期——从创建到结束，经历哪些状态，每个状态能做什么。

```yaml
# business/sales/workflow.yaml
workflow:
  name: 销售订单流程
  model: sale.order

  states:
    - name: draft
      label: 草稿
      allowed_actions: [确认, 取消]
      who_can_operate: [销售员, 销售经理]

    - name: confirmed
      label: 已确认
      allowed_actions: [提交审批, 取消]
      who_can_operate: [销售员, 销售经理]
      auto_actions:
        - condition: 订单金额 <= 公司审批阈值
          action: 自动跳过审批，直接进入已审批

    - name: pending_approval
      label: 待审批
      allowed_actions: [审批通过, 审批拒绝]
      who_can_operate: [销售经理, 销售总监]
      timeout: 48小时无人审批则通知销售总监

    - name: approved
      label: 已审批
      allowed_actions: [创建发货单]
      who_can_operate: [销售员, 销售经理]

    - name: done
      label: 已完成
      allowed_actions: []
      is_final: true

    - name: cancelled
      label: 已取消
      allowed_actions: [重新打开]
      who_can_operate: [销售经理]

  transitions:
    - from: draft → to: confirmed → action: action_confirm
    - from: confirmed → to: pending_approval → action: action_submit_approval
    - from: confirmed → to: approved → condition: 无需审批
    - from: pending_approval → to: approved → action: action_approve
    - from: pending_approval → to: draft → action: action_reject
    - from: approved → to: done → action: action_done
    - from: [draft, confirmed] → to: cancelled → action: action_cancel
    - from: cancelled → to: draft → action: action_reopen
```

#### 类型 C：数据模型（第三优先）

字段定义、关联关系、计算公式——描述"你的业务世界长什么样"。

```yaml
# business/sales/models.yaml
models:
  - name: 销售审批单
    technical_name: sale.approval
    description: 记录销售订单的审批过程

    fields:
      - name: order_id
        label: 关联订单
        type: many2one
        target: sale.order
        required: true

      - name: approver_id
        label: 审批人
        type: many2one
        target: res.users
        default: 当前用户的上级

      - name: approval_amount
        label: 审批金额
        type: monetary
        compute: 取自关联订单的总金额

      - name: decision
        label: 审批结果
        type: selection
        options: [待审批, 同意, 拒绝]
        default: 待审批

      - name: reason
        label: 审批意见
        type: text
        required_when: 审批结果 = 拒绝
```

#### 类型 D：权限与安全（第四优先）

谁能看什么、谁能做什么——这些规则也是业务知识的一部分。

```yaml
# business/sales/security.yaml
security:
  groups:
    - name: 销售员
      can_do: [查看自己的订单, 创建订单, 确认订单]
      can_see: [订单基本信息, 客户信息]
      cannot_see: [成本价, 利润率]

    - name: 销售经理
      inherits: 销售员
      can_do: [查看所有订单, 审批订单, 取消订单]
      can_see: [成本价, 利润率]

    - name: 销售总监
      inherits: 销售经理
      can_do: [修改审批阈值, 查看统计报表]

  record_rules:
    - name: 销售员只看自己的订单
      applies_to: 销售员
      condition: 订单的销售员 = 当前用户
```

### 为什么用 YAML 而不是用数据库

有人可能会问：为什么不把业务规则存到数据库的配置表里？

| | YAML 文件 | 数据库配置表 |
|---|---|---|
| 版本控制 | Git 天然支持 | 需要额外导出/导入 |
| 代码审查 | PR 里看 diff 就行 | 看不到变化 |
| 环境同步 | Git pull 就同步了 | 需要手动同步或写脚本 |
| AI 可读性 | AI 直接读文件 | 需要查数据库 |
| 可迁移性 | 复制文件即可 | 需要导出/转换 |
| 人类可读性 | 直接打开就看 | 需要后台界面 |

**简单说：YAML 文件跟代码一起走，享受代码管理的一切好处（版本控制、审查、回滚、分支）。**

当然，某些需要运行时动态修改的配置（比如"审批阈值"的具体数字）更适合放在数据库的系统参数或配置面板里。YAML 存的是"规则结构"，数据库存的是"运行时参数"。两者配合使用。

## 五、怎么一步步走到那里

不要一步到位。分三步，每步都有独立价值。

### 第一步：方法拆小（现在已经在做）

这就是 Agent.md 里要求的"业务逻辑与 ORM 分离"。

**之前：** 一个大方法，什么都混在一起
```python
def action_confirm(self):
    if self.amount_total > 100000:  # 业务规则
        raise UserError("需要审批")   # Odoo 写法
    self.write({'state': 'confirmed'})  # Odoo 写法
    self.env['stock.picking'].create({...})  # Odoo 写法
```

**之后：** 拆成小方法，业务规则独立
```python
def action_confirm(self):
    self._check_can_confirm()        # 业务规则（独立方法）
    self._do_confirm()               # Odoo 写法（薄包裹）
    self._create_delivery()          # Odoo 写法（薄包裹）

def _needs_approval(self):
    """纯业务规则：这个方法将来可以直接提取出去"""
    return self.amount_total > self.company_id.approval_threshold
```

**价值：** 代码更清晰，重构更容易。将来提取业务规则时，`_needs_approval()` 可以直接搬走。

**具体拆法——四种方法前缀：**

| 前缀 | 含义 | 依赖 Odoo？ | 未来可提取？ |
|---|---|---|---|
| `_needs_*()` | 业务判断：需不需要做某事 | 尽量不依赖 | 最容易提取 |
| `_check_*()` | 业务校验：能不能做某事 | 尽量不依赖 | 容易提取 |
| `_prepare_*()` | 准备数据：组装字典 | 依赖字段名 | 较容易提取 |
| `_do_*()` | 执行操作：写数据库 | 重度依赖 | 不需要提取（是适配层） |

**这一步的验收标准：**

- 每个 `action_*` 方法不超过 10 行
- 每个业务判断都有独立的 `_needs_*` 或 `_check_*` 方法
- 纯业务判断方法里没有 `self.env`、`self.sudo()`、`raise UserError`
- 所有 ORM 操作（create / write / unlink / search）集中在 `_do_*` 和 `_prepare_*` 方法里

### 第二步：业务规则外置（重构时做）

把散落在代码里的业务判断，提取成配置文件：

```yaml
# business/sales/approval_rules.yaml
rules:
  - name: high_value
    condition: "order_total > config.approval_threshold"
    action: require_approval

  - name: new_customer_first_order
    condition: "customer.is_new and customer.order_count == 0"
    action: require_approval
    approver: sales_director
```

然后写一个简单的"规则引擎"来读取它：

```python
def _needs_approval(self):
    # 不再硬编码业务规则
    # 而是从配置文件里读取
    return self.env['rule.engine'].evaluate(
        'sales.approval', self
    )
```

**价值：** 业务规则集中管理，改规则不需要改代码。

**规则引擎不需要很复杂：**

最简单的版本大概长这样（伪代码）：

```python
class RuleEngine(models.AbstractModel):
    _name = 'rule.engine'

    def evaluate(self, rule_path, record):
        """读取 YAML 规则文件，逐条评估"""
        rules = self._load_rules(rule_path)
        for rule in rules:
            if self._eval_condition(rule['condition'], record):
                return self._exec_action(rule['action'], record)
        return None

    def _load_rules(self, path):
        """从 business/ 目录读取 YAML 文件，带缓存"""
        # 开发时从文件读取
        # 生产环境可以启动时加载到内存
        ...

    def _eval_condition(self, condition_str, record):
        """安全地评估条件表达式"""
        # 用预定义的安全函数集合评估
        # 不要用 eval()！
        ...
```

**这里有一个重要的安全原则：绝不能用 `eval()` 来执行条件表达式。** 应该把条件表达式解析成一棵语法树，只允许预定义的安全操作（比较、逻辑运算、字段访问）。这其实和 Odoo 自己的 domain 表达式是同一个思路。

**这一步的验收标准：**

- 所有业务规则都有对应的 YAML 配置文件
- YAML 文件由不懂代码的产品经理也能看懂
- 修改业务规则只需要改 YAML，不需要改 Python
- 规则变更可以通过 Git PR 审查

### 第三步：事件驱动解耦（更远的将来）

用"广播 + 订阅"替代 Odoo 的"覆写"方式：

现在的方式——销售模块和库存模块紧密耦合：
```
销售模块 ──_inherit──> 库存模块
（销售模块直接修改了库存模块的代码）
```

将来的方式——通过事件通讯，互不依赖：
```
销售模块 ──广播"订单已确认"──> 事件总线 ──通知──> 库存模块（创建发货单）
                                        ──通知──> 财务模块（创建应收单）
                                        ──通知──> 通知模块（发邮件）
                                        ──通知──> 任何将来的模块
```

**事件驱动的好处：**

1. **加新功能不改旧代码**：想在订单确认后发一条微信通知？加一个"订阅者"就行，不用碰销售模块的代码。
2. **删功能不怕连锁反应**：不需要微信通知了？直接删掉订阅者，其他模块完全不受影响。
3. **升级时互不影响**：每个模块只关心自己的事件，不关心别的模块怎么实现的。
4. **天然支持异步**：发货单创建可以放到后台队列，不阻塞订单确认。

**事件定义也是 YAML 配置：**

```yaml
# business/events.yaml
events:
  - name: order_confirmed
    label: 订单已确认
    source: sale.order
    payload:
      - order_id: 订单ID
      - partner_id: 客户ID
      - amount_total: 订单金额
      - company_id: 公司ID

  - name: order_cancelled
    label: 订单已取消
    source: sale.order
    payload:
      - order_id: 订单ID
      - cancel_reason: 取消原因

# business/subscriptions.yaml
subscriptions:
  - event: order_confirmed
    subscriber: stock_module
    action: 创建发货单
    async: true

  - event: order_confirmed
    subscriber: account_module
    action: 创建应收单
    async: false  # 财务必须同步完成

  - event: order_confirmed
    subscriber: notification_module
    action: 发送确认邮件
    async: true
```

**Odoo 其实已经有事件机制的雏形：**

Odoo 的 `mail.thread` mixin 的 `message_post`、`tracking` 等功能就是一种简化的事件系统。我们要做的不是从零开始，而是把这个机制扩展为通用的业务事件总线。

**这一步的验收标准：**

- 模块间不再通过 `_inherit` 互相引用业务逻辑
- 添加新的"响应模块"不需要修改"触发模块"的代码
- 事件的发布和订阅都在 YAML 里配置
- 每个事件都有明确的 payload 定义

## 六、什么不应该提取

不是所有东西都适合从代码里"拔"出来。以下几种情况应该保留在 Odoo 代码里：

### 6.1 ORM 操作本身

`create()`、`write()`、`search()` 这些是 Odoo 提供的基础能力，不是业务知识。它们就应该留在代码里作为"适配层"。

```python
# 这是适配层代码，不需要提取
def _do_confirm(self):
    self.write({'state': 'confirmed', 'confirm_date': fields.Datetime.now()})
```

### 6.2 纯 UI 布局

字段在页面上放在左边还是右边、用什么颜色、标签叫什么——这些是 UI 决策，不是业务知识。它们应该留在 Odoo 的视图 XML 里。

### 6.3 性能优化逻辑

缓存、批量处理、SQL 优化——这些是技术实现细节，跟业务无关。提取出来反而丧失了优化的灵活性。

### 6.4 框架级集成

与 Odoo 框架深度集成的功能（安装、卸载、模块依赖、数据文件加载顺序）应该留在 `__manifest__.py` 和标准数据文件里。

### 判断标准

> **这个逻辑换了 ERP 软件还需要吗？**
>
> - 需要 → 这是业务知识，应该提取
> - 不需要 → 这是技术实现，留在代码里

## 七、这跟我们现有的工具链是什么关系

我们的工具链（Agent.md + DEVELOPMENT_STANDARD.md + pre-commit + PR Review）不是独立于这个愿景的，它是**第一步的执行工具**。

```
架构愿景（本文）
    ↓ 指导
开发规范（DEVELOPMENT_STANDARD.md）
    ↓ 精简为
AI 指令（Agent.md）
    ↓ 执行时
代码检查（pre-commit + PR Review）
    ↓ 保障
代码质量 → 为第二步、第三步打基础
```

### 具体对应关系

| 愿景要求 | 工具链里的落地 |
|---|---|
| 业务逻辑拆成独立方法 | Agent.md §3："业务判断提成 `_needs_*()` / `_check_*()` 独立方法" |
| 不侵入官方代码 | Agent.md §1："不修改 Odoo 官方源码"、"不复制整份视图 XML" |
| 一个概念一个权威来源 | DEVELOPMENT_STANDARD §2.2："同一业务概念只能有一个权威来源" |
| 方法可被 override | Agent.md §3："复杂流程拆成可 override 的小方法链" |
| 批量优先 | Agent.md §1："不在循环内执行 search"（保证适配层性能） |
| 安全默认 | Agent.md §1："不用 sudo() 包裹整个方法" |

**换句话说：如果你严格按照 Agent.md 写代码，你产出的代码天然就是"第一步完成"的状态。** 这就是为什么我们把方法拆分的要求放在日常开发规范里——不是为了好看，是为了将来能走到第二步和第三步。

## 九、怎么衡量成功

做了这些投入，怎么知道有没有效果？以下是每一步的关键指标：

### 第一步的成功指标

- **方法平均行数 < 10 行**：说明拆分充分
- **纯业务判断方法占比 > 30%**：说明业务逻辑确实被分离出来了
- **PR Review 通过率提升**：说明代码质量在提高
- **新人/AI 上手时间缩短**：小方法更容易理解

### 第二步的成功指标

- **改业务规则不需要改 Python 文件**：核心目标
- **产品经理能直接审查规则变更的 PR**：说明 YAML 足够可读
- **规则变更的上线时间从"天"降到"小时"**：说明流程简化了
- **业务规则覆盖率**：有多少 `_needs_*` / `_check_*` 方法已经被 YAML 替代

### 第三步的成功指标

- **模块间 `_inherit` 业务方法的数量趋近于零**：说明解耦充分
- **添加新的响应模块不需要修改现有模块代码**：核心目标
- **Odoo 版本升级的工作量减少 50% 以上**：最终 ROI 指标

## 十、风险与应对

任何架构变革都有风险。提前认识到风险，才能控制它。

### 风险 1：过度设计

**症状：** 为了一个只有 3 条规则的小模块，搭建了一套完整的规则引擎。

**应对：** 遵循"三个以上再抽象"原则。如果某类规则只有 1-2 条，直接写在代码里完全没问题。等到第 3 条出现时再考虑提取。

### 风险 2：规则引擎变成黑盒

**症状：** YAML 规则越来越复杂，支持嵌套条件、循环引用、动态计算……最终 YAML 变成了另一种编程语言，反而更难维护。

**应对：** 严格限制 YAML 的表达能力。条件只支持简单比较和逻辑运算。如果一条规则用 YAML 写出来比 Python 还复杂，那它就应该留在 Python 里。

### 风险 3：性能退化

**症状：** 规则引擎每次都要读文件、解析 YAML、评估条件，比直接写 if-else 慢。

**应对：** 加缓存。YAML 文件在模块加载时一次性读入内存，运行时不再读文件。业务规则的执行频率很低（一笔订单确认一次），即使有额外开销也可以忽略不计。真正的性能问题从来都是数据库查询，不是业务判断。

### 风险 4：调试困难

**症状：** 业务逻辑从代码"搬"到了 YAML，出 bug 时不知道是规则写错了还是引擎解释错了。

**应对：**
- 规则引擎必须有日志：每次评估规则时记录输入、匹配到的规则、输出结果
- YAML 规则必须有单元测试
- 开发环境提供"规则调试模式"：可以看到每条规则的评估过程

### 风险 5：团队抵触

**症状：** 开发者觉得"我写 Python 挺好的，为什么要搞 YAML"。

**应对：** 不要强推。从一个痛点明显的场景开始（比如审批规则经常变），让大家看到效果后自然接受。第一步（方法拆小）是几乎零成本的——它只是更好的代码组织方式，不需要任何额外工具。

## 十一、常见问题

### Q：这是不是过度设计？

如果你只有几个简单模块，永远不升级 Odoo，那确实不需要。

但如果你面临以下任何一种情况，就值得投入：
- Odoo 版本升级频繁或即将到来
- 模块数量超过 10 个
- 多人 / 多 AI 并行开发
- 业务规则经常变化
- 有可能将来换 ERP 或同时使用多个系统
- 合规要求需要留存业务规则的变更记录

### Q：第二步的"规则引擎"复杂吗？

最简单的版本就是一个读取 YAML 文件并执行条件判断的函数，可能不到 100 行代码。不需要做成什么复杂的商业规则引擎。

如果你熟悉 Odoo 的 domain 表达式（`[('state', '=', 'draft')]`），规则引擎的条件评估本质上跟它是一回事——只是换了一种更可读的写法。

### Q：这样做性能会不会差？

不会。业务规则执行频率很低（一笔订单确认一次），多一层配置文件读取的开销可以忽略。真正影响性能的是数据库查询，不是业务判断。

实测数据参考：解析一个包含 50 条规则的 YAML 文件大约需要 1-2 毫秒。一次数据库查询通常需要 5-50 毫秒。规则评估的开销连数据库查询的零头都不到。

### Q：AI 能帮我做这件事吗？

可以。事实上 AI 特别适合做这件事：
- 让 AI 读现有代码，提取出业务规则，写成 YAML
- 让 AI 根据 YAML 生成 Odoo 模块代码
- 让 AI 在 Odoo 升级时，对比新旧版本的 API 差异，自动更新适配层
- 让 AI 审查 YAML 规则文件的完整性和一致性

这也是我们坚持"业务逻辑拆成独立方法"的原因——方法越独立越纯粹，AI 越容易识别和提取。

### Q：能不能跳过第一步，直接做第二步？

**强烈不建议。** 第一步是第二步的基础。如果业务逻辑还混在大方法里，你不知道哪些是业务规则、哪些是 ORM 操作。先拆分，才能看清楚什么需要提取。

类比：你要搬家（第二步），但东西还都堆在一个大箱子里。你得先分类整理（第一步），才知道哪些要带走、哪些可以扔。

### Q：第三步（事件驱动）一定要做吗？

不一定。第三步的收益主要体现在大型项目（20+ 模块、多团队并行开发）。如果你的项目规模不大，做完前两步就已经能解决大部分问题。

第三步更像是"终极形态"——如果你做到了，恭喜，你的 Odoo 项目已经比大多数企业的 ERP 系统架构更先进了。

### Q：现有的代码怎么迁移？

不需要一口气迁移。采用"绞杀者模式"（Strangler Fig Pattern）：

1. **新代码**按新规范写（方法拆小、规则外置）
2. **改旧代码**时顺带重构（触碰到哪个方法就拆哪个）
3. **不主动重构**没有变更需求的旧代码

这样，随着时间推移，新规范的代码会逐渐"包裹"并替代旧代码，就像绞杀榕慢慢包裹宿主树一样。

### Q：这个思路适用于其他框架吗？

完全适用。"业务知识与技术实现分离"是一个通用的架构原则，不限于 Odoo。如果你还在用 Django、Rails、Spring Boot 或其他任何框架，同样的思路都能帮你降低升级成本和技术锁定风险。

## 十二、一句话总结

> **业务知识是你公司最值钱的东西，不应该被锁死在任何一个软件的代码里。**

我们现在做的每一步——方法拆小、规则分离、事件解耦——都是在一点一点把业务知识从 Odoo 的代码里"拔"出来，放到一个任何人、任何 AI、任何系统都能理解和使用的地方。

**不要等到升级痛了才动手。从今天写的每一行代码开始，用"将来能不能提取"的标准来要求自己。** 这就是我们这套工具链存在的意义。
